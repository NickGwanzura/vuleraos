# Phase 3: Zimbabwe-Specific Features — Implementation Plan

## Scope
Three feature areas that make VuleraOS genuinely Zimbabwean: **Multi-Currency Engine**, **ZIMRA Compliance**, and **Mobile Money Payments**. Plus a **Settings** module to configure them.

---

## 1. Multi-Currency Engine

### What Exists
- `Currency` model (USD, ZWG with isBase flag)
- `ExchangeRate` model (rate, parallelMarketRate, effectiveDate, isManualOverride)
- Currency-aware fields on items, invoices, POs, payments

### What We Need

#### Exchange Rate Management UI (`/settings/currency`)
- Daily exchange rate entry: rate + parallel market rate
- History view showing rate changes over time
- Manual override toggle
- Quick "copy parallel to official" button for days when the gap matters

#### API Routes
| Route | Methods | Purpose |
|---|---|---|
| `/api/settings/currency/rates` | GET, POST | List exchange rates (with date filter), create new rate |
| `/api/settings/currency/rates/latest` | GET | Get latest rate for a currency pair |

#### Dual-Currency Display Utility
A helper function/library that converts between USD and ZWG amounts using the stored exchange rate. Display both currencies side-by-side on:
- Invoice detail page (show ZWG equivalent for USD invoices)
- Dashboard revenue metrics
- Reports

#### UI Features
- Exchange rate list: date, rate, parallel rate, override indicator, who set it
- Quick entry form: date picker, two rate fields, override checkbox
- History chart (simple): line showing rate trend over last 30 days

---

## 2. ZIMRA Compliance

### What Exists
- `FiscalDevice` model
- `isFiscalised`, `fiscalReceiptNumber` fields on invoices
- `bpNumber` (BP/VAT number) on tenants and business partners
- `vatRate`, `vatAmount` on invoices

### What We Need

#### Fiscal Device Management (`/settings/fiscal-devices`)
- Register and manage fiscal devices (TIN/BP number, serial number, model)
- Track device communication status

#### ZIMRA-Compliant Invoicing
- **VAT calculation**: Auto-calculate at 15% standard rate, allow per-line override (zero-rated, exempt)
- **BP Number**: Require on customers before fiscal invoice can be issued
- **Fiscalisation**: "Issue Fiscal Invoice" button in invoice detail that:
  1. Validates the invoice has all required fields
  2. Marks as fiscalised with receipt number
  3. Transitions status from DRAFT → FISCAL
  4. (Future: sends to FDMS)

#### ZIMRA Reports
- **VAT Return Report**: Summary of output VAT (sales) and input VAT (purchases) for a period
- Downloadable CSV

#### API Routes
| Route | Methods | Purpose |
|---|---|---|
| `/api/settings/fiscal-devices` | GET, POST | List/register fiscal devices |
| `/api/sales/invoices/[id]/fiscalise` | PUT | Issue invoice as fiscal (validate + mark) |
| `/api/reports/vat-return` | GET | VAT return data for a date range |

#### UI Features
- Fiscal device list + registration form
- "Issue as Fiscal" button on invoice detail (changes status, assigns receipt number)
- VAT return report page in `/reports` module

---

## 3. Mobile Money Payments

### What Exists
- `PaymentMethod` enum includes `ECOCASH`, `ONEMONEY`
- `referenceNumber` field on payments

### What We Need

#### Payment Method Display & Filtering
- Enhanced payment forms with:
  - EcoCash: transaction ID field (validated format: `EC\d+`)
  - OneMoney: transaction ID field
  - RTGS: bank name + reference field
  - Bank Transfer: bank name dropdown (Zimbabwe bank list)

#### Mobile Money Quick Entry on Invoices
- "Receive EcoCash Payment" button on invoice detail
- Opens dialog: enter transaction ID + amount + sender phone
- Creates payment and marks invoice as paid/partially paid

#### API Routes
| Route | Methods | Purpose |
|---|---|---|
| `/api/accounting/payments/mobile` | POST | Record mobile money payment with validation |

#### UI Features
- Mobile money-specific dialog on invoice detail
- Payment method labels with icons
- Reference number format hints
